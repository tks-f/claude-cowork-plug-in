# Indeed 求人分析スキル

## 概要

Indeed Employer Dashboardから全求人のパフォーマンスデータを収集し、競合分析と合わせてExcelレポートを出力する。求人改善の優先順位付けと具体的なアクション提案まで行う。

## 前提条件

- Claude in Chrome（ブラウザ自動操作）が利用可能
- Indeed Employer Dashboard（employers.indeed.com）にログイン済み
- xlsxスキルを併用してExcelファイルを生成する（SKILL.mdを最初に読むこと）

## ワークフロー

全体は4フェーズで進行する：データ収集 → 競合調査 → 改善提案生成 → Excel出力

### Phase 1: パフォーマンスデータ収集

Indeed Employer Dashboardの求人一覧・詳細ページからKPIデータを収集する。

#### 1-1. 求人一覧の取得

1. `employers.indeed.com` の求人一覧ページへ遷移
2. 「募集中」ステータスの求人をすべてリストアップ
3. 各求人について取得する情報：
   - 求人タイトル
   - 掲載日（ページ上の「○日前」や日付表記から推定）
   - ステータス（募集中 / 一時停止 / 終了）

#### 1-2. 各求人のKPI取得

各求人の詳細ページ（Job Details）にアクセスし、「パフォーマンスの概要」セクションから読み取る：

- **表示回数**（Impressions）
- **クリック数**（Clicks）
- **応募数**（Applications）
- **データ対象期間**（例：2026年2月1日〜2026年2月18日）

ページ読み込み後2〜3秒待ってからスクリーンショットまたはget_page_textで取得する。IndeedはReact SPAなので即座にはデータが表示されないことがある。

数値が「-」や空欄の場合は0として記録する。

#### 1-3. KPI算出

| KPI | 計算式 | 何を測っているか |
|-----|--------|----------------|
| CTR（クリック率） | クリック数 ÷ 表示回数 × 100 | タイトル・給与・条件が求職者の目を引くか |
| 応募率 | 応募数 ÷ クリック数 × 100 | 詳細ページが応募を後押しするか |
| 応募CV率 | 応募数 ÷ 表示回数 × 100 | 全体の採用効率 |

#### 1-4. 改善優先度スコアリング

各求人に0〜100の改善優先度スコアを付ける。スコアが高い＝改善が急務で効果も大きい。

判定ロジック（上から順に適用、最初に該当した条件でスコア確定）：

| 条件 | スコア | 理由 |
|------|--------|------|
| 表示 ≥ 50 かつ CTR = 0% | 95 | 見られているのに全くクリックされない＝タイトル/給与に深刻な問題 |
| 表示 ≥ 50 かつ CTR < 3% | 80 | 表示は十分だがクリックが少ない＝タイトル/条件に改善余地 |
| CTR ≥ 3% かつ 応募率 = 0% | 75 | クリックはされるが応募ゼロ＝詳細ページの訴求力が不足 |
| CTR ≥ 3% かつ 応募率 < 5% | 60 | 詳細ページに改善余地 |
| 表示 < 20 | 30 | そもそも見られていない＝まず露出改善が先（リフレッシュ等） |
| CTR ≥ 5% かつ 応募率 ≥ 10% | 10 | 好調。現状維持でOK |
| それ以外 | 50 | 中程度の改善余地 |

### Phase 2: 競合・市場分析

自社求人ごとに、Indeedの求職者向け検索ページで類似求人を調査する。

#### 2-1. 競合求人の検索

1. `jp.indeed.com`（求職者向けページ）にアクセス
2. 自社求人の職種名＋勤務地で検索
3. 上位5〜10件の競合求人から収集：
   - 求人タイトル
   - 企業名
   - 給与（表示されていれば）
   - 雇用形態
   - 目立つキーワード・特徴（リモート、フレックス等）

#### 2-2. 市場データの整理

| 分析項目 | 内容 |
|---------|------|
| 給与レンジ | 競合の最低〜最高給与 |
| 給与中央値 | 競合給与の中央値 |
| 自社ポジション | 自社給与が市場の上位/中位/下位のどこか |
| 頻出キーワード | 競合タイトルによく出るワード |
| 不足キーワード | 競合にあって自社タイトルにないワード |

給与が非公開の競合は「非公開」と記録し集計からは除外する。

### Phase 3: 改善提案と具体的な改善コンテンツの生成

Phase 1のKPIとPhase 2の競合データを掛け合わせて、求人ごとの具体的アクションを提案する。
**さらに、各改善項目について「そのまま使える改善後のコンテンツ」を生成する。**

#### 3-1. 改善アクションの特定

提案は以下のカテゴリで整理する：

| 問題パターン | 改善カテゴリ | アクション例 |
|------------|-----------|------------|
| CTRが低い | タイトル改善 | 競合で効果的なキーワード追加、文字数最適化 |
| CTRが低い | 給与改善 | 市場中央値以上に引き上げ、給与レンジを明記 |
| CTRが低い | 条件面改善 | リモート・フレックスをタイトルに追記 |
| 応募率が低い | 仕事内容改善 | 冒頭にポイントまとめ追加、具体的な業務内容の記載 |
| 応募率が低い | 必須条件緩和 | 経験年数の引き下げ、歓迎スキルへの移動 |
| 表示が少ない | 露出改善 | 求人リフレッシュ、キーワードSEO最適化 |

各アクションに「難易度」（簡単/中/難）と「期待インパクト」（高/中/低）を付ける。

#### 3-2. 具体的な改善コンテンツの生成

各求人について、以下の改善コンテンツを**具体的なテキストとして**生成する。
現在の求人内容＋競合分析データ＋KPI問題パターンに基づき、実際に掲載可能なレベルの文面を作成する。

**① タイトル改善案（2〜3案）**

生成ルール：
- 競合で頻出のキーワードを自然に組み込む
- 「リモート」「フレックス」「未経験OK」等の訴求ワードがあれば追加
- 文字数は全角30文字以内を目安（Indeed検索結果で途切れないように）
- 現在のタイトルとの変更点を明記

例：
```
現在：API基盤エンジニア（Java/Spring）
改善案1：API開発エンジニア（Java/Spring Boot）リモート週2可
改善案2：【Java】API基盤開発エンジニア｜月給45万〜・リモートOK
改善案3：Spring Boot × API開発エンジニア｜上流工程から参画
```

**② 給与改善案**

生成ルール：
- 競合の給与中央値を参考に、市場競争力のある金額を提案
- 「月給○○万円〜○○万円」のレンジ表記を推奨
- 現在の給与からの引き上げ幅と理由を記載

例：
```
現在：月給350,000円
改善案：月給400,000円〜550,000円
根拠：競合中央値が月給42万円。上限を明示することで高スキル層にもアピール。
```

**③ 仕事内容の改善案（全文）**

生成ルール：
- 冒頭に「▍ この求人のポイント」として3〜5行の箇条書きサマリーを配置
- 「携わる業務」セクションで具体的な業務内容を記述
- 「技術スタック・環境」セクションで使用技術を整理
- 箇条書きは「・」を使い、各項目は1〜2行で簡潔に
- 競合にあって自社にない訴求ポイントを補完

**④ アピールポイントの改善案（全文）**

生成ルール：
- 5項目以内、各項目は「◆ 」で始まる1行の簡潔な文
- 競合との差別化ポイントを最初に持ってくる
- 具体的な数字や条件を盛り込む（「週2リモート可」「年間休日120日以上」等）

**⑤ 求める人材の改善案（全文）**

生成ルール：
- 「【必須スキル】」「【歓迎スキル】」「【求める人物像】」の3セクション構成
- 必須スキルは最小限（2〜3項目）に絞る → 応募のハードルを下げる
- 現在の必須条件が厳しすぎる場合は歓迎スキルに移動
- 求める人物像は「〜な方」のポジティブ表現で3〜4項目
- 経験年数は競合平均と比較して適切な水準に設定

#### 3-3. Before/After比較

各改善コンテンツについて「現在の内容」と「改善案」を並べて比較できるようにする。
変更箇所にはマーク（★）を付けて、何がどう変わったかを一目で分かるようにする。

### Phase 4: Excelレポート出力

**xlsxスキルを必ず読み込むこと**。SKILL.mdのパスは以下を確認：
`/sessions/jolly-vibrant-hamilton/mnt/.skills/skills/xlsx/SKILL.md`

#### シート構成

**シート1「サマリー」**
全求人のKPI一覧を改善優先度スコア降順でソート。

| 列 | 内容 |
|----|------|
| A | 求人タイトル |
| B | 掲載日 |
| C | 表示回数 |
| D | クリック数 |
| E | 応募数 |
| F | CTR (%) |
| G | 応募率 (%) |
| H | 優先度スコア |

書式ルール：
- CTR 0% → セル背景を赤
- CTR ≥ 5% → セル背景を緑
- 優先度スコア ≥ 80 → 行背景を薄い赤
- ヘッダー行は濃い青背景＋白文字
- データ集計期間をシート上部に明記

**シート2「競合分析」**
求人ごとの競合比較。

| 列 | 内容 |
|----|------|
| A | 自社求人タイトル |
| B | 競合企業名 |
| C | 競合求人タイトル |
| D | 競合給与 |
| E | 自社給与 |
| F | 給与差分 |
| G | 市場ポジション |

**シート3「アクションプラン」**
改善アクションを優先度順にリスト。

| 列 | 内容 |
|----|------|
| A | 優先順位 |
| B | 対象求人 |
| C | 問題パターン |
| D | アクション内容 |
| E | 難易度 |
| F | 期待インパクト |

**シート4「改善コンテンツ（タイトル・給与）」**
求人ごとのタイトル改善案と給与改善案を一覧表示。

| 列 | 内容 |
|----|------|
| A | 対象求人（現タイトル） |
| B | タイトル改善案1 |
| C | タイトル改善案2 |
| D | タイトル改善案3 |
| E | 現在の給与 |
| F | 改善後の給与案 |
| G | 給与改善の根拠 |

書式ルール：
- 改善案のセルは薄い黄色背景で目立たせる
- 各求人は1行で完結
- ヘッダー行は濃い青背景＋白文字

**シート5「改善コンテンツ（仕事内容）」**
求人ごとの仕事内容の改善案を全文掲載。

| 列 | 内容 |
|----|------|
| A | 対象求人タイトル |
| B | 現在の仕事内容（全文） |
| C | 改善後の仕事内容（全文） |
| D | 主な変更点（★マーク付き箇条書き） |

書式ルール：
- B列・C列は「折り返して全体を表示」に設定
- 列幅はB・Cとも60文字程度に広げる
- C列（改善後）は薄い黄色背景
- 行の高さは内容に合わせて自動調整

**シート6「改善コンテンツ（アピール・求める人材）」**
求人ごとのアピールポイントと求める人材の改善案を全文掲載。

| 列 | 内容 |
|----|------|
| A | 対象求人タイトル |
| B | 現在のアピールポイント |
| C | 改善後のアピールポイント |
| D | 現在の求める人材 |
| E | 改善後の求める人材 |
| F | 主な変更点（★マーク付き箇条書き） |

書式ルール：
- B〜E列は「折り返して全体を表示」に設定
- 列幅は各50文字程度に広げる
- C・E列（改善後）は薄い黄色背景
- 行の高さは内容に合わせて自動調整

#### ファイル名

`indeed_analytics_YYYYMMDD.xlsx`（実行日の日付）

出力先：`/sessions/jolly-vibrant-hamilton/mnt/outputs/`

## データ取得時の注意

- Indeed Employer DashboardはReact SPAなので、ページ遷移後に2〜3秒待つ
- スクリーンショットとget_page_textを組み合わせると正確にデータが取れる
- 求人一覧が複数ページにわたる場合はページネーションで全件取得する
- 分析はスナップショット（実行時点のデータ）であり、時系列追跡はしない
- レポートには必ずデータ取得日時と対象期間を記載する
